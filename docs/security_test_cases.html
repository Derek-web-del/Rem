How to Execute Security Test Cases

Before every test session 
cd c:\xampp\htdocs\LenLearn
npm run dev

API port 3001, Vite 5173. Use incognito for auth tests. If 429 errors: restart dev server.

Optional: npm run security:evidence → outputs in docs/evidence/automated/

AUTH-001 — Multiple failed login attempts
  1. Open http://localhost:5173/login/student with a test student account.
  2. Enter correct username + wrong password 5 times.
  3. 6th attempt should still fail (account locked ~5 minutes).
  4. Check audit: SELECT * FROM lms_activity_logs WHERE "activityType"='AUTH_LOCKOUT' ORDER BY "timestamp" DESC LIMIT 1;
Mark Passed when: Lockout triggered after 5 attempts. Screenshot login error + audit row.
AUTH-002 — Invalid credentials
  1. Try valid username + wrong password → note error.
  2. Try fake username + any password → note error.
Mark Passed when: Same generic error both times (no user enumeration).
AUTH-003 — Protected page without login
  1. Incognito → visit /admin/institute_dashboard, /student/dashboard, /teacher/dashboard → all redirect to login.
  2. Run: curl -s -i http://127.0.0.1:3001/api/v1/student/terms-status → HTTP 401.
Mark Passed when: Redirect successful + API 401.
AUTH-004 — Session expiration (idle)
  1. Sign in, stay idle 30+ min with tab visible → auto sign-out.
  2. Quick test: Temporarily set IDLE_MS = 60000 in StudentLayout.jsx, wait 1 min idle, revert after.
Mark Passed when: Client idle sign-out after 30 minutes. Tell panel: server cookie is 7 days.
AUTH-005 — Wrong portal login
  1. Faculty login URL + student credentials → rejected.
  2. Student login URL + faculty credentials → rejected.
Mark Passed when: Access blocked at wrong portal.
SESS-001 — Logout invalidates session
  1. Sign in → Logout → try /student/dashboard → login required.
  2. Or curl protected API after logout → 401.
Mark Passed when: Session invalidated; 401 after logout.
ACCESS-001 / ACCESS-002 — RBAC redirects
  1. ACCESS-001: Student signed in → go to /teacher/dashboard → redirected to student dashboard.
  2. ACCESS-002: Faculty signed in → go to /admin/institute_dashboard → redirected to teacher dashboard.
Mark Passed when: 100% blocked cross-role access.
ACCESS-003 — Unauthenticated API 
curl -s -i http://127.0.0.1:3001/api/v1/student/terms-status
Mark Passed when: HTTP 401 + sign-in required message.
INPUT-001 / INPUT-002 — File upload validation
  1. INPUT-001: Faculty → upload .jpg or .exe to PDF-only study materials → rejected.
  2. INPUT-002: Upload file >10 MB (student) or >25 MB (materials) → size error.
INPUT-003 — XSS test
  1. Enter <script>alert('xss')</script> in announcement or name field.
  2. Save and reopen → text shows literally, no alert popup.
SQLI-001 — Login SQL injection
  1. Username: ' OR '1'='1 Password: ' OR '1'='1
  2. Login must fail with no SQL error in response.
CONFIG-002 — Security headers 
curl -s -I http://127.0.0.1:3001/health

Look for: content-security-policy, x-content-type-options: nosniff, x-frame-options: SAMEORIGIN

LOG-001 / LOG-002 — Audit logging
  1. Failed login + successful login → Admin → Audit Logs → events visible.
  2. Faculty saves grade → audit row with old/new values.
DATA-001 / DATA-003 — Encryption & password hash 
-- PII encryption
SELECT LEFT(first_name,15), LEFT(last_name,15) FROM students ORDER BY id DESC LIMIT 3;

-- Password hash
SELECT LEFT(a.password,7) FROM account a WHERE a."providerId"='credential' LIMIT 3;

PII should show enc:v1:. Password should show $2b$12.
